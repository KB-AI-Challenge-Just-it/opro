## 1. 문제 정의

소상공인은 상권 변화, 금리 변동, 정책자금 공고 등 경영에 직접 영향을 주는 정보를 놓치기 쉽다. 정보가 없는 게 아니라 **흩어져 있고, 변화가 생겼을 때 알려주는 주체가 없다**는 게 핵심 문제다.

기존 서비스는 대부분 사용자가 검색·질문해야 답을 주는 반응형 구조다. 이 프로젝트는 에이전트가 상황을 상시 감시하다 먼저 개입하는 **능동형 구조**를 목표로 한다.

### 1-1. 기존 KB국민은행 소상공인 관련 서비스 현황

조사 결과, KB국민은행은 이미 소상공인 대상 서비스를 여러 채널로 운영 중이다.

| 서비스 | 형태 | 내용 |
| --- | --- | --- |
| 상권분석시스템 (KB부동산 Liiv ON) | 온라인 리포트 | 반경 500m 기준 상권분석보고서·업종분석보고서 제공 |
| 우리가게 매출순위·신용도 (KB스타기업뱅킹) | 셀프 진단 | KB국민카드 가맹점 매출 데이터 기반 매출 순위·요일별/시간대별 매출 비중 자가 확인 |
| KB 소호 컨설팅센터 / 소상공인 One-Stop 컨설팅센터 | 오프라인 상담 | 상권분석·자금조달·세무·SNS 등 무료 컨설팅, 전국 13개 센터 운영 |
| KB상권활성화지수 | B2B/B2G 분석 모델 | 금융데이터 기반 상권 진단 지수 — 지자체·공공기관·민간기업 대상 |
| 리브 넥스트 AI금융비서 | 일반 AI 챗봇 | Z세대 대상 범용 금융 비서, 소상공인 특화 아님 |

### 1-2. 기존 서비스의 한계

- **정적·1회성 리포트**: 상권분석시스템은 사용자가 조회를 요청한 시점의 스냅샷만 제공한다. 조회 이후 상권에 변화가 생겨도 별도로 알려주지 않는다.
- **자가진단 의존**: 우리가게 매출순위·신용도는 사용자가 직접 앱에 접속해 확인해야 하는 구조다. 이상 신호가 생겨도 능동적으로 알려주는 알림 기능은 확인되지 않는다.
- **오프라인·사람 의존적 컨설팅**: 소호 컨설팅센터는 전국 13개 센터의 예약 기반 상담으로 운영되어, 상시 모니터링이나 실시간 대응이 구조적으로 불가능하다.
- **B2B/B2G 중심 분석**: KB상권활성화지수는 지자체·공공기관·민간기업을 위한 분석 도구로, 개별 소상공인이 직접 사용하는 서비스가 아니다.
- **정책자금 자동 매칭 부재**: 경쟁사(하나은행)는 이미 AI 기반 정책자금 맞춤 조회 서비스를 운영 중인 것으로 확인되나, KB는 개별 소상공인 대상 자동 매칭 서비스가 뚜렷하게 확인되지 않는다.
- **데이터·채널 분절**: 상권(Liiv ON) · 매출(스타기업뱅킹) · 정책자금(컨설팅센터)이 각각 다른 채널에 흩어져 있어, 사용자가 스스로 여러 곳을 오가며 정보를 종합해야 한다.

### 1-3. 이 서비스가 해결하는 방향

| 기존 KB 서비스의 한계 | 본 서비스의 해결 방향 |
| --- | --- |
| 조회 시점 스냅샷만 제공 (상권분석시스템) | 상시 모니터링 + 임계값 기반 자동 트리거로 변화 발생 시점에 능동 알림 |
| 사용자가 직접 확인해야 하는 자가진단 (매출순위·신용도) | 에이전트가 먼저 이상 신호를 감지해 리포트를 push |
| 오프라인 예약 상담, 확장성 제한 (소호 컨설팅센터) | AI 파이프라인으로 상시·자동 대응, 인력 의존도 최소화 |
| 지자체·기관 대상 분석 도구 (상권활성화지수) | 개별 소상공인이 직접 받아보는 개인화 리포트로 전환 |
| 정책자금 매칭 서비스 부재 | RAG 기반으로 상황에 맞는 정책자금을 근거와 함께 자동 매칭 |
| 상권·매출·정책자금 채널 분절 | 4개 데이터 축을 하나의 파이프라인으로 통합, 단일 리포트로 제공 |

**핵심 포지셔닝**: KB는 이미 상권분석·매출진단·컨설팅·정책자금 안내 각각의 요소를 개별적으로 보유하고 있다. 이 프로젝트의 차별점은 새로운 데이터를 만드는 것이 아니라, **흩어진 기존 기능들을 "상시 감시 + 자동 트리거 + 통합 판단"이라는 에이전틱 구조로 엮어내는 것**이다. 이는 대출 상담 연계 등 KB의 기존 비즈니스 모델과도 자연스럽게 이어질 수 있다.

## 2. 목표 및 성공 기준

| 구분 | 내용 |
| --- | --- |
| 개발 목표 | 1개 업종 × 1개 지역 시나리오에서 "감지 → 분석 → 정책자금 매칭 → 제안"이 실제 데이터로 end-to-end 동작 |
| 성공 기준 (MVP) | ① 4개 데이터소스 중 최소 3개 실제 연동 ② 트리거 발동 → Claude 원인 분석 → 정책자금 매칭까지 자동 파이프라인 완주 ③ 결과 리포트가 사람이 읽고 납득 가능한 수준 |
| 확장 가능성 증명 | 업종/지역 코드를 파라미터로 교체 가능한 구조인지 코드 레벨에서 확인 (하드코딩 여부 점검) |

MVP 범위는 의도적으로 좁게 잡는다. 넓게 벌리다 일부가 목업으로 채워지는 것보다, 좁고 확실하게 도는 것이 개발 우선순위다. (근거: 데모/검증 신뢰도 확보, 4주 일정 내 현실성)

## 4. 서비스 흐름 (유저 시나리오)

1. 사장님이 최초 1회 **온보딩 폐쇄형 질문지**(4-1 참고)에 답변해 사업자 프로필 등록 — 자유 서술형이 아닌 선택형으로 설계해 응답률 확보
2. 에이전트가 백그라운드에서 4개 데이터소스를 주기적으로 모니터링
3. 임계값 초과 이벤트 발생 (예: 반경 500m 내 동일업종 신규 개업 3건 이상, 기준금리 변동 등)
4. 에이전트가 자동으로 원인 분석 실행, 관련 정책자금 매칭
5. 사장님에게 "지금 상황 – 원인 – 대응 가능한 정책자금" 리포트 push
6. **매칭된 정책자금에 대해 신청서류 초안을 자동 생성** (5-3 참고) — 사장님은 검토·수정 후 직접 제출
7. (확장 시나리오) 사장님이 리포트 내에서 바로 정책자금 신청 페이지로 연결 / 대출 상담 연계

## 5. 시스템 아키텍처

### 5-1. 데이터 축

| 축 | 소스(안) | 역할 | 확보 난이도 |
| --- | --- | --- | --- |
| 상권 | 소상공인시장진흥공단 상권정보 API | 경쟁강도, 유동인구, 매출추이 | 중 (공공API, 신청 필요) |
| 경기지표 | 한국은행 ECOS, KOSIS | 금리, 물가, 업종별 BSI | 낮음 (공개 API) |
| 소비 트렌드 | 카드사 매출 통계 | 지역·업종별 소비 변화 | **높음** (세부 데이터 접근 제한 가능성) |
| 정책자금 | 기업마당, 소상공인정책자금 공고 | 신청 가능 지원사업 매칭 | 중 (크롤링/API 혼용 가능) |

카드사 소비 데이터는 접근이 막힐 가능성이 높으므로, MVP에서는 **상권 + 경기지표 + 정책자금 3축**을 우선 확보하고 소비 트렌드는 대체 지표(예: 상권 API 내 매출추이로 갈음)로 처리하는 것을 기본 시나리오로 잡는다.

### 5-2. 파이프라인

```
① 온보딩 → ② 원인 분석(Claude) → ③ 정책자금 매칭(RAG) → ④ 리포트 생성 → ⑤ 신청서류 초안 생성 → ⑥ 자동 알림
```

- 온보딩 : 앞서 말했던 4-1. 온보딩 질문지 (폐쇄형 설계)를 통해 정보를 얻는다.
- 원인 분석: Claude API에 다축 데이터 컨텍스트를 넣어 다단계 추론
- 정책자금 매칭: 정책자금 공고문 임베딩 후 벡터 검색 (RAG) — 단순 키워드 필터보다 "왜 맞는지" 근거 제시 가능
- 리포트 생성: 사장님이 이해하기 쉬운 언어로 최종 출력 (텍스트 우선, 여유 있으면 시각화 확장)
- 신청서류 초안 생성: 매칭된 정책자금의 신청 양식에 맞춰 초안을 자동 작성 (5-3 참고)
- 자동 알림: 처음 온보딩을 한번 수행해두면, polling을 하면서 사용자에게 능동적으로 알림을 주는 어시스턴트로 동작한다.

### 5-3. 신청서류 초안 자동 생성 (RAG 확장 기능)

**목적**: 지금까지는 "이 정책자금이 있다"는 정보 전달에서 그쳤다면, 여기서는 "신청서 초안까지 이미 써놨다"로 한 단계 더 나아가 실제 신청 행동까지 이어지게 만든다.

**입력**:

- 매칭된 정책자금의 공고문 원문 + 신청서 양식(요구 항목 구조)
- 4-1 온보딩 질문지 응답값 (사업자 프로필)
- L3에서 생성된 원인 분석 텍스트 (경영 상황 설명)

**처리**: Claude(Sonnet)가 공고문에서 신청서 요구 항목(예: 사업 개요, 신청 사유, 지원받고자 하는 내용, 기대 효과)을 파악하고, 온보딩 프로필과 원인 분석 결과를 근거로 각 항목의 초안 문장을 작성한다.

**출력**: 편집 가능한 신청서 초안 문서. **사장님이 검토·수정 후 직접 제출**하는 것을 전제로 한다 — 정부 포털 인증·실제 제출은 사용자 본인 책임 영역으로 남기고, 자동 제출 기능은 만들지 않는다.

**기술적 리스크 — 반드시 인지할 것**: 정부지원사업 신청서 양식은 상당수가 hwp·pdf 첨부파일로 공고문 밖에 별도 제공되는 경우가 많다. 이 양식을 텍스트로 구조화하는 난이도가 공고마다 다를 수 있어, 전체 정책자금에 일괄 적용하기보다 **MVP 데모 시나리오에 쓰는 1~2개 공고에 한해 양식을 미리 확보해 시범 적용**하는 것을 권장한다.

**일정 반영**: 기존 RAG 개발 일정(3~5일)에 그대로 흡수하기보다, 3주차 end-to-end 완주를 선행 조건으로 두고 **4주차 초반에 +1~2일 별도 작업으로 배치**하는 것을 추천한다.
