# 온보딩 매장 검색 방식: 소진공 시/군구 전체 스캔 (juso.go.kr 주소검색·카카오 로컬 API 보류)

**상태**: 결정 완료 (소진공 상가업소정보 API, 시/군구 단위 전체 페이지네이션 스캔)

**관련 문서**: `doc/onboarding.md` 화면1, `doc/상가정보_OpenApi문서.pdf`, 이슈 #32

---

## 1. 문제 정의

`doc/onboarding.md` 화면1 "사장님 가게를 찾아볼게요"는 상호명으로 매장을 검색하는 기능을 전제한다.
그런데 실제로 연동한 소진공(소상공인시장진흥공단) 상가업소정보 API(B553077)는
`doc/상가정보_OpenApi문서.pdf` 목차의 오퍼레이션 19개(지정/반경/사각형/다각형/행정구역단위 상권조회,
단일/건물/지번/행정동/상권내/반경내/사각형내/다각형내/업종별/수정일자기준 상가업소조회 등) 중
**상호명으로 검색하는 오퍼레이션이 하나도 없다.** 전부 좌표·행정구역·건물·PNU·업종 기반 "목록 조회"고,
"이 이름을 가진 게 어디 있나"를 묻는 인터페이스가 아니다.

초기 구현(`SbizStoreSearchClient`)은 시/도 전체를 `storeListInDong`으로 1페이지(1000건)만 받아
상호명을 서버에서 `contains` 필터링했는데, 실사용 중 "이천우화"라는 실제 대구 중구 소재 상호를
찾지 못하는 문제가 발생했다. 원인 조사 결과:
- 대구 중구만 해도 `totalCount=14,681`건 — 1페이지(1000건)로는 애초에 커버 불가
- 전체 14,681건을 다 받아 확인해도 "이천우화"는 데이터셋 자체에 없었음(분기별 갱신·등록명 상이 등 추정)

즉 문제는 두 겹이었다: (a) 이름검색 오퍼레이션 부재, (b) 페이지네이션 미구현으로 1000건 밖은 아예 못 봄.
(b)는 고칠 수 있지만 (a)는 이 API 패밀리의 구조적 한계라 근본 해결이 안 된다.

## 3. 결정: 소진공 시/군구 전체 스캔

**"이름으로 완전 검색"은 이번 스코프에서 포기한다.** 대신:

1. 화면1에 **시/군구 드롭다운을 추가**한다(기존엔 시/도만 있었음) — `GET /api/onboarding/sigungu?sido=` 신규 엔드포인트, 소진공 `baroApi`(catId=cty)로 조회.
2. 선택된 시/군구의 상가업소 목록을 `storeListInDong`(divId=signguCd)으로 **페이지네이션 전체 순회**(최대 20페이지=2만 건, `SbizStoreSearchClient.searchInSigungu`)해서 상호명 `contains` 필터링한다.
3. 검색 UX엔 "구 전체를 뒤지는 중이라 최대 수십 초 걸릴 수 있다"는 안내를 명시하고, 결과가 없으면 "직접 입력하기"를 동급 경로로 제공한다 — 이 데이터셋에 아예 없는 매장은 어떤 방식으로도 못 찾기 때문에, "검색"을 메인이 아니라 "직접 입력"의 보조 경로로 취급한다.

juso.go.kr 주소검색(②)과 카카오 로컬 키워드검색(③)은 기술적으로 더 나은 대안이지만, 이번 판단은
**새 API 계정을 늘리지 않고 기존 흐름을 유지하는 쪽을 택한 것**이다(1인 개발 MVP, 이미 발급된 SBIZ 키 재사용).
"소상공인만 걸러야 한다"는 이유는 재검토 결과 근거가 약함을 확인했다 — 세 API 모두 사업장 규모로
필터링하지 않으며, 이 기능 자체가 필터링을 요구하지 않는다(사용자가 검색하는 대상은 본인 가게).
