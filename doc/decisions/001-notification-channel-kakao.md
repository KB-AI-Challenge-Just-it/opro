# 알림 채널 의사결정 기록: 웹 실시간 알림에서 카카오톡 알림까지

**문서 목적**: 알림 전달 채널을 검토한 배경·과정·결론을 남겨, 이후 구현·데모 준비·확장 논의 때 같은 고민을 반복하지 않기 위한 개발 기록

**상태**: 결정 완료 (인앱 폴링 = P1 필수, 카카오 나에게 보내기 = P1.5 데모 강화, 알림톡 = 확장 로드맵)

**관련 문서**: DEPENDENCY_GRAPH.md (의존성 그래프 내 알림 트랙)

---

## 1. 문제 정의 — 왜 알림 트랙이 필요해졌나

초기 의존성 그래프는 파이프라인이 "리포트 생성·저장"에서 끝나 있었다.
그런데 이 서비스의 1번 차별화는 **"사용자가 물어보기 전에 먼저 알려주는 능동형 구조"**다.
스케줄러가 매일 돌면서 신규 공고를 감지하고 프로필과 매칭까지 해도,
**그 결과가 사용자에게 도달하는 경로**가 없으면 차별화가 성립하지 않는다.

정리하면 요구사항은 세 가지였다.

1. 배치(매일 06:00)가 신규 공고를 감지 → 매칭 → 리포트 생성까지 완료하면
2. 해당 사용자에게 알림이 **능동적으로 도달**해야 하고
3. 사용자는 알림을 받아 리포트 상세로 진입할 수 있어야 한다

여기서 중요한 설계 인식 하나:
**"감지"는 새 컴포넌트가 아니다.** 신규 공고 감지는 이미 수집기(pblanc_id 기준 델타 upsert) →
트리거 엔진 → 매칭 파이프라인이 하고 있는 일이다. 추가로 필요한 것은
`PipelineService.run()` 마지막에 **notification 레코드를 insert하는 한 스텝**과,
그 레코드를 사용자에게 **전달하는 채널**뿐이다.

이에 따라 DB 스키마(P0)에 notification 테이블이 추가되었다:

```sql
CREATE TABLE notification (
  id         BIGSERIAL PRIMARY KEY,
  profile_id BIGINT REFERENCES business_profile(id),
  report_id  BIGINT REFERENCES report(id),
  title      TEXT NOT NULL,
  read_at    TIMESTAMPTZ,          -- null이면 미확인
  created_at TIMESTAMPTZ DEFAULT now()
);
```

> ⚠️ 이 테이블은 P0 스키마 확정에 반드시 포함할 것. 나중에 넣으면 마이그레이션 작업이 하나 늘어난다.

## 2. 검토 과정 — 웹 실시간 알림에서 시작한 이유

### 2-1. 1차 후보: 웹 인앱 알림 (폴링 vs SSE vs WebSocket)

처음에는 웹 화면 안에서 알림을 보여주는 방식부터 검토했다.
비교 기준은 "우리 알림의 발생 빈도"였다.

| 방식 | 동작 | 판단 |
| --- | --- | --- |
| **폴링** | 프론트가 5~10초 간격으로 `GET /api/notifications` 호출 | ✅ MVP 필수로 채택 |
| **SSE** | 서버가 열린 연결로 이벤트 push (Spring `SseEmitter`) | 데모 강화 후보 (반나절 작업) |
| **WebSocket** | 양방향 상시 연결 | ❌ 제외 — 양방향이 필요 없음 |

핵심 논거: **배치가 하루 1회 도는 구조라 알림 발생 시점 자체가 하루 한 번이다.**
밀리초 단위 실시간(WebSocket)은 명백한 과투자이고, 폴링이면 사용자 경험상 충분하다.
SSE는 데모에서 "수동 트리거 실행 → 화면에 알림이 즉시 뜨는" 장면을 극적으로
만들 수 있어 강화 옵션으로 남겼다.

### 2-2. 방향 전환의 계기: "소상공인은 카톡에 산다"

웹 알림의 구조적 한계가 논의되었다.

- **인앱 알림은 접속해야 보인다.** 사장님이 우리 웹을 매일 열어본다는 가정은
  "사용자가 직접 확인해야 하는 자가진단" — 즉 우리가 기존 KB 서비스의 한계라고
  비판한 바로 그 구조로 되돌아가는 것이다.
- 능동형 서비스라면 알림이 **사용자의 생활 채널로 찾아가야** 한다.
- 타겟(소상공인 사장님)의 생활 채널은 앱 푸시도 이메일도 아닌 **카카오톡**이다.
  주문·예약·거래처 연락이 전부 카톡에서 일어난다.

또한 데모 임팩트 차이가 컸다. SSE 토스트는 "웹앱이면 다 하는 것"으로 보이지만,
수동 트리거를 실행하고 **무대 위 휴대폰이 실제로 울리는 장면**은
"능동형 에이전트"라는 컨셉을 3초 만에 증명한다.

### 2-3. 결론된 구조: 웹 알림 폐기가 아니라 계층화

카카오로 전환하되 웹 알림을 버리지 않는다. 이유는 리스크 분리다.

```
[P1  · 필수]   인앱 폴링 알림  — 데모의 필수 경로. 외부 의존 없음, 실패 불가
[P1.5 · 강화]  카카오 나에게 보내기 — 데모 클라이맥스. 실패해도 데모가 죽지 않음
[P3  · 로드맵] 알림톡/KB 인프라 — 실서비스 단계에서 승계
```

> 원칙: **데모의 필수 경로에 외부 API를 두지 않는다.**
> 카카오 API 장애·토큰 만료·시연장 네트워크 문제가 발생해도
> 인앱 알림으로 데모 시나리오가 완주 가능해야 한다.

## 3. 카카오톡 알림의 선택지 비교

"카카오톡으로 알림을 보낸다"에는 실제로 3가지 다른 제품이 있고, 난이도가 크게 다르다.

### 용어 정리 먼저

- **웹훅이 아니다**: 웹훅은 카카오 → 우리 서버 방향(수신)이다. 우리가 필요한 것은
  우리 서버 → 사용자 카톡 방향(발신)이며, 이는 카카오 비즈메시지/메시지 API 계열이다.

### 3-1. 알림톡 (비즈메시지)

| 항목 | 내용 |
| --- | --- |
| 방식 | 카카오톡 채널 명의로 발송되는 정보성 메시지 |
| 요건 | 채널 개설 → **사업자등록번호 기반 비즈니스 인증** → 딜러사(솔라피·NHN Cloud·알리고 등) 계약 → **템플릿 사전 심사 (영업일 2~3일)** |
| 비용 | 건당 약 7~15원 |
| 장점 | 도달률 최강, 친구 추가 불필요, 실서비스 정석 |
| 단점 | 심사·인증이 개발 밖의 대기 시간. 템플릿 문구 수정 시마다 재심사. 팀 명의 사업자 없으면 진행 불가 |
| 판단 | ❌ MVP 제외 — 4주 일정에서 리스크 대비 이득 없음. **확장 로드맵으로**: "실서비스에서는 KB 알림 인프라 또는 알림톡으로 승계" |

### 3-2. 친구톡 / 채널 메시지

| 항목 | 내용 |
| --- | --- |
| 방식 | 채널 친구에게 발송하는 메시지 (광고성 포함 가능) |
| 요건 | 알림톡과 동일한 채널·딜러사 체계 + 수신자가 채널 친구여야 함 + 광고성 수신동의 |
| 판단 | ❌ 제외 — 알림톡 대비 나은 점이 없고 제약만 추가 |

### 3-3. 카카오톡 "나에게 보내기" API ⭐ 채택

| 항목 | 내용 |
| --- | --- |
| 방식 | 사용자가 카카오 OAuth로 `talk_message` 권한 동의 → 우리 서버가 **그 사용자 본인의 카톡방(나와의 채팅)**으로 메시지 발송 |
| 요건 | 카카오 디벨로퍼스 앱 등록만. **사업자 인증 불필요, 템플릿 심사 불필요, 무료** |
| 엔드포인트 | `POST https://kapi.kakao.com/v2/api/talk/memo/default/send` |
| 제약 | 그 계정 **본인에게만** 발송 가능 |
| 판단 | ✅ 채택 — 데모에서는 페르소나 계정 하나면 충분하므로 "본인에게만"이라는 제약이 오히려 정확히 들어맞음 |

## 4. 채택안 상세 설계 — 나에게 보내기

### 4-1. 흐름

```
[온보딩]
사장님 → "카카오톡으로 알림 받기" 버튼
  → 카카오 OAuth 동의 (scope: talk_message)
  → 액세스 토큰 + 리프레시 토큰 발급
  → Spring이 profile에 토큰 저장

[배치 알림 발송]
ScheduledJobs (06:00)
  → PipelineService.run()
     → 리포트 저장
     → notification insert          ← 인앱 알림 (P1 필수)
     → KakaoMemoSender.send()       ← 카톡 발송 (P1.5, 실패해도 파이프라인 정상)
```

### 4-2. API 호출 형태

```
POST https://kapi.kakao.com/v2/api/talk/memo/default/send
Authorization: Bearer {사용자 액세스 토큰}
Content-Type: application/x-www-form-urlencoded

template_object={
  "object_type": "text",
  "text": "[신규 공고] 사장님 해당 1건 발견 — 마감 D-5\n경기 소상공인 특례보증 지원사업",
  "link": { "web_url": "https://<우리도메인>/reports/123" },
  "button_title": "리포트 확인"
}
```

### 4-3. 필요한 스키마 변경

```sql
-- 카카오 토큰 저장 (business_profile 확장 또는 별도 테이블)
CREATE TABLE kakao_token (
  profile_id    BIGINT PRIMARY KEY REFERENCES business_profile(id),
  access_token  TEXT NOT NULL,
  refresh_token TEXT NOT NULL,
  expires_at    TIMESTAMPTZ NOT NULL,
  refreshed_at  TIMESTAMPTZ DEFAULT now()
);
```

### 4-4. 작업량 추정

| 작업 | 규모 |
| --- | --- |
| 카카오 디벨로퍼스 앱 등록 + Redirect URI 설정 | 1시간 |
| OAuth 동의 → 토큰 저장 플로우 (Spring) | 0.5일 |
| KakaoMemoSender (WebClient 발송 + 토큰 갱신) | 0.5일 |
| 온보딩 UI에 동의 버튼 (web) | 0.5일 |
| **합계** | **Spring 1일 + web 0.5일** |

## 5. 개발 이슈 및 대응 — 반드시 읽을 것

### 5-1. 토큰 수명 관리 (가장 중요한 운영 함정)

- 카카오 **액세스 토큰은 수 시간 단위로 만료**된다. 배치는 하루 뒤에 도는데
  온보딩 때 받은 액세스 토큰은 그 시점에 이미 죽어 있다.
- 따라서 **발송 직전 리프레시 토큰으로 액세스 토큰을 갱신하는 로직이 필수**다.
  "온보딩 때 토큰 저장 → 다음날 그대로 사용"은 100% 실패하는 설계다.
- 리프레시 토큰도 만료가 있다(약 2개월, 갱신 시 연장). MVP 기간에는 문제없지만
  코드에 갱신 실패 → 인앱 알림만 발송 + 재동의 유도 플래그 처리를 넣어둘 것.

```
발송 시퀀스:
  1. expires_at 확인 → 만료(임박)면 리프레시로 갱신 후 DB 업데이트
  2. 발송 시도
  3. 401 응답이면 갱신 1회 재시도 → 그래도 실패면 카톡 스킵, 인앱 알림만
```

### 5-2. 발송 실패가 파이프라인을 죽이면 안 된다

- `KakaoMemoSender.send()`는 **try-catch로 감싸고 실패를 로그만 남긴다.**
  카톡 발송 실패로 notification insert나 리포트 저장이 롤백되는 트랜잭션 설계 금지.
- 순서 강제: **notification insert가 먼저, 카톡 발송이 나중.**
  (인앱이 원본, 카톡은 미러라는 계층 구조를 코드에도 반영)

### 5-3. OAuth 설정 함정

- **Redirect URI 정확 일치**: 카카오는 등록된 URI와 한 글자만 달라도
  (`http` vs `https`, 포트, 트레일링 슬래시) `KOE006` 에러. 로컬(`http://localhost:3000/...`)과
  데모 환경 URI를 **둘 다 미리 등록**해둘 것.
- **동의항목 활성화**: 디벨로퍼스 콘솔에서 `talk_message`(카카오톡 메시지 전송)
  동의항목을 켜야 scope 요청이 동작한다. 기본 off라 빠뜨리기 쉬움.
- 앱이 "테스트 앱" 상태면 **팀 멤버로 등록된 카카오 계정만** 로그인 가능.
  데모 페르소나 계정을 팀 멤버로 미리 추가해둘 것. (비즈 앱 전환은 MVP에 불필요)

### 5-4. 메시지 형식 제약

- `template_object`는 **URL 인코딩된 form 파라미터**로 보내야 한다. JSON body로
  보내면 400. (Spring WebClient에서 `BodyInserters.fromFormData` 사용)
- text 템플릿 본문 최대 길이 제한이 있으므로(약 200자) 리포트 전문이 아니라
  **제목 + D-day + 링크**만 담고 상세는 웹 리포트로 유도한다. 어차피 이게 UX상 정답.
- `link.web_url`은 디벨로퍼스에 등록된 **플랫폼 도메인과 일치**해야 버튼이 동작한다.
  로컬 데모면 localhost 등록 필요.

### 5-5. 데모 리스크 체크리스트

- [ ] 데모 당일 아침 토큰 갱신 확인 (리허설 체크리스트에 포함)
- [ ] 시연장 네트워크에서 kapi.kakao.com 접근 확인 (방화벽 이슈 대비)
- [ ] 카톡 실패 시 인앱 알림으로 시나리오 완주 가능한지 리허설 1회
- [ ] 페르소나 카카오 계정을 앱 팀 멤버로 등록했는지 확인
- [ ] 무대 폰의 카톡 알림음 켜져 있는지 (마지막에 웃지 못할 함정)

### 5-6. 한계 인지 — 실서비스와의 갭

"나에게 보내기"는 본인 계정에게만 발송되므로 **불특정 다수 사용자 서비스로는
확장 불가**하다. 이는 결함이 아니라 의도된 단계 설계다:

| 단계 | 채널 | 근거 |
| --- | --- | --- |
| MVP 데모 | 나에게 보내기 + 인앱 폴링 | 심사 없음, 무료, 1일 구현 |
| 실서비스 | KB 앱 push 또는 알림톡 | KB 탑재 전제 — 인증·알림 인프라 승계 |

발표 Q&A에서 "이거 본인한테만 가는 거 아니냐"는 질문이 나오면 위 표가 답변이다.
알림 채널은 어댑터 하나 교체(카카오 → KB push)로 전환되도록
`NotificationSender` 인터페이스 뒤에 구현을 숨겨둘 것.

| 결정 | 내용 |
