# 003: 기동 준비 상태 노출 + 온보딩 결과 동기화 여부

## 배경

두 가지 사용자 경험 문제가 반복적으로 관측됐다.

**문제 1 — `docker compose up` 직후 바로 못 씀.** ai-engine은 임베딩 모델(bge-m3)을
모듈 임포트 시점에 동기 로드하는데, 콜드 캐시일 때 최대 몇 분 걸린다(디스크가
빠듯하면 더 걸리거나 실패). PR #65(merged)로 `ai-engine`에 Docker healthcheck를
추가하고 `api-core`의 `depends_on`을 `condition: service_healthy`로 바꿔서,
**`docker compose up` 명령 자체는** ai-engine이 healthy할 때까지 기다렸다가 리턴한다.
즉 CLI에서 그 명령을 실행한 사람은 이미 "다 기다린 뒤"에야 프롬프트를 돌려받는다.

그런데 이걸로 못 막는 경우가 남는다: **web 컨테이너는 `api-core`가 시작해야
같이 시작하는 구조라(`depends_on: [api-core]`), 콜드스타트가 도는 동안은
3000번 포트 자체가 안 열려 있다.** 그 시간에 브라우저로 접속하면 "연결 거부"만
보이고, 기다리라는 안내조차 뜰 곳이 없다 — 서버가 없으니 화면을 못 띄운다.
(compose 명령을 실행한 사람 본인은 이미 안내를 받은 셈이지만, 그 사람이
아닌 다른 사용자가 그 사이 브라우저로 먼저 들어가면 못 막는다.)

**문제 2 — 온보딩 결과가 "됐다"고 떴는데 실제로는 안 된 것처럼 보임.**
`OnboardingController.submit()`은 프로필+웰컴 리포트만 동기로 저장하고,
실제 매칭·분석·리포트 생성(Claude 3회 호출)은 가상 스레드로 비동기 실행한다
(이슈 #53). 프론트(`onboarding/page.tsx`)는 `GET /api/onboarding/{id}/match-status`를
1.5초마다 폴링하다가 **60초가 지나면 실제 완료 여부와 무관하게 그냥 완료 화면으로
넘어간다**(`setMatchSettled(true)`). 즉 파이프라인이 60초보다 오래 걸리면
사용자는 "됐다"는 화면을 보지만 실제로는 백그라운드에서 계속 처리 중이거나
막 실패했을 수 있다 — 이게 지금 "결과가 안 나온다"고 느끼는 원인 중 하나다.

## 확인된 사실 (추측 아님)

- PR #65는 main에 **merge됨**(2026-07-21 08:48 KST). 인프라 레벨 게이팅은 이미 적용돼 있다.
- `MatchStatusTracker`는 순수 인메모리(`ConcurrentHashMap`)다 — api-core가 재시작되면
  진행 상태가 사라진다(DB에 영구 저장 안 함). 영구화하려면 별도 테이블이 필요하다.
- api-core → ai-engine 호출(`AiEngineClient`)은 호출 1건당 240초 타임아웃이다.
  matching·analysis·report 3단계를 순차로 부르면 이론상 최대 12분까지 걸릴 수 있지만,
  실제로는 Claude 호출이 보통 10~30초(코드 주석 기준)라 정상 케이스는 훨씬 짧다.
  Spring MVC·Tomcat 쪽에 별도 요청 타임아웃 설정은 없다 — 오래 걸려도 서버가
  스스로 끊지는 않는다. 프론트의 `api()`(fetch)도 별도 타임아웃을 안 걸어서,
  브라우저가 요청을 계속 열어두는 것 자체는 기술적으로 문제 없다.

## 문제 1: 기동 준비 상태를 프론트에서 인지 가능하게

### 지금 상태로 충분한 경우
"컴포즈를 실행한 그 사람"만 신경 쓴다면 PR #65로 이미 해결이다 — 명령이 안 끝났으면
그 사람은 아직 브라우저를 열 이유가 없다. **문서(README)에 "첫 실행은 최대 5분
걸릴 수 있고, 터미널에 `ai-engine ... Healthy`가 찍힌 뒤 접속하라"는 안내만 추가하면
코드 변경 없이 끝난다.**

### 실제 "플로우로 만들기"를 원한다면 (권장)
web을 api-core의 헬스 체인에서 분리해서 **먼저 뜨게** 하고, 프론트 자체가
"백엔드 준비 중" 화면을 띄우면서 준비될 때까지 기다리게 한다.

1. `docker-compose.yml`: `web`의 `depends_on`을 없애거나 `postgres`만 두고
   `api-core` 의존을 뺀다 — web 컨테이너(Next.js 서버)가 백엔드 상태와 무관하게 즉시 뜨게.
2. `api-core`에 가벼운 readiness 엔드포인트 추가 (예: `GET /api/health`).
   api-core 자체가 ai-engine healthy 이후에만 기동하므로(#65), 이 엔드포인트가
   응답한다는 것 자체가 ai-engine까지 정상이라는 뜻 — 별도로 ai-engine을
   다시 확인할 필요 없음.
3. 프론트 최상단(`layout.tsx` 또는 별도 게이트 컴포넌트)에서 앱 렌더 전에
   이 엔드포인트를 2~3초 간격으로 폴링. 실패(연결 거부 포함)하면
   "시스템을 준비하고 있어요 (최대 5분 정도 걸릴 수 있어요)" 스피너를 보여주고,
   성공하면 그제서야 실제 화면을 그린다.

**트레이드오프**: compose 파일과 신규 엔드포인트, 신규 프론트 컴포넌트가 필요하다
(작업량 있음). 대신 "누가 언제 브라우저를 열든" 항상 올바른 안내를 받는다.

### 결정 필요
- A) 문서만 추가 (당장 코드 변경 없음, 대신 "compose 실행한 사람 = 접속하는 사람"이
  아닐 때는 여전히 연결 거부를 봄)
- B) web 분리 + readiness 폴링 프론트 게이트 (완전한 해결, 작업량 있음)

## 문제 2: 비동기 → 동기, 오래 걸려도 사용자가 인지

**지금 코드의 실제 문제**: 60초가 지나면 실제 완료 여부와 무관하게 프론트가
"완료" 화면으로 강제 전환한다. 이게 거짓 완료 신호를 준다 — 사용자가 지적한
"결과가 실제로 안 만들어졌다"는 인식이 바로 이 지점에서 나온다.

리포트 히스토리(`/profiles/[id]`)와 알림(notification)이 이미 있어서, 탭을 닫거나
새로고침해도 결과 자체는 나중에 어디서든 다시 볼 수 있다 — "탭을 닫으면
결과를 영영 놓친다"는 걱정은 이미 어느 정도 완화돼 있다.

### 옵션 A — 진짜 동기로 바꾸기 (사용자가 원한 방향)
`POST /api/onboarding`이 매칭·분석·리포트 생성까지 다 끝난 뒤 응답한다.
프론트는 그 요청이 도는 동안 "처리 중입니다... (N초 경과)" 스피너만 보여주면 되고,
그 스피너가 떠 있는 시간 = 실제로 걸리는 시간이라 **거짓말을 할 수가 없다.**
구현은 오히려 단순해진다 — `Thread.ofVirtual().start()` 제거하고 그 자리에서
바로 `profileMatchTrigger.runForProfile()`을 호출하면 됨. `MatchStatusTracker`,
`/match-status` 폴링, 스텝퍼 UI도 전부 걷어낼 수 있다.

- 장점: 설계가 단순해짐. "처리 중" 표시가 실제 시간과 정확히 일치.
- 단점: 요청 하나가 최악의 경우 몇 분간 열려있음. 그 사이 브라우저 탭을
  닫으면 그 요청 자체의 응답은 못 받지만(HTTP 연결 끊김), 서버 쪽 처리는
  이미 시작됐으면 계속 진행되고 리포트/알림은 정상 생성된다 — 사용자는
  나중에 알림이나 질문 목록에서 확인 가능.

### 옵션 B — 비동기 유지, "포기하는 타임아웃"만 제거
지금 구조(폴링 + 스텝퍼)를 유지하되, 60초 후 강제 전환하는 로직만 없앤다.
완료될 때까지 계속 폴링하면서 "검색 중 → 분석 중 → 리포트 생성 중"처럼
단계별 라벨을 계속 보여준다. 실제로 안 끝났으면 화면도 안 끝난 것처럼 보인다 —
거짓말을 안 하게 되는 건 A와 동일하지만, 단계별 진행 표시는 유지된다.

- 장점: 단계별 진행률("지금 몇 단계") 표시 가능. 탭을 계속 열어놔도
  HTTP 요청 하나가 몇 분씩 안 열려 있어도 됨(가벼움).
- 단점: 코드는 그대로 두 갈래(즉시 응답 + 별도 폴링 API)라 A보다 구조가 복잡함.
  `MatchStatusTracker`가 인메모리라 그 사이 api-core가 재시작되면 진행 상태가
  끊겨서 폴링이 "검색 중"으로 리셋돼 보일 수 있음(드묾, 로컬 개발 환경 한정 이슈).

### 결정 필요 — 이 질문 하나로 A/B가 갈림
**"검색 중 → 분석 중 → 생성 중"처럼 단계별로 뭘 하고 있는지 보여주고 싶은가,
아니면 그냥 "처리 중입니다"만 정직하게 보여주면 충분한가?**
- 단계별 표시가 필요하다 → **옵션 B**(폴링 유지, 타임아웃만 제거)
- "처리 중"만으로 충분하다 → **옵션 A**(진짜 동기, 코드 단순화)
