# Dev Orchestrator — 작업 선택 · 게이트 강제 · 레인 위임

1인 개발자가 AI 하네스로 이 MVP를 구현할 때의 조율 스킬. 얇게 유지한다 — 실행 지식은 각 에이전트에 있고, 작업 현황은 GitHub 이슈로 추적한다(구 `doc/work_breakdown01.md`는 2026-07-21 삭제). 여기는 **작업 선택과 게이트 강제**만 담는다.

## 실행 모드: 서브 에이전트

크리티컬 패스(S4→S5→S6)가 순차라 팀 통신이 불필요하다. 병렬은 **직교 작업에만** background 최대 2개 (예: backend-dev의 S5 ∥ web-dev의 W2 mock 선행).

## 에이전트 구성

| 에이전트 | subagent_type | 레인 (편집 허용 범위) | 참조 스킬 |
| --- | --- | --- | --- |
| backend-dev | `backend-dev` | `apps/api-core/**`, `db/**` | — |
| ai-dev | `ai-dev` | `apps/ai-engine/**` | rag-conventions |
| web-dev | `web-dev` | `apps/web/**` | — |
| qa-verifier | `qa-verifier` | 편집 없음 (검증 리포트만) | e2e-verify |

모든 Agent 호출에 `model: "opus"`를 명시한다.

## 위임 임계 (delegate vs inline)

- **위임**: GitHub 이슈 단위 — 레인 스코프의 기능 구현
- **인라인(메인 스레드 직접)**: 단일 파일 소규모 수정, 문서 갱신, 설정값 변경 — 스폰 비용이 이득을 초과

## 워크플로우

### Phase 0: 컨텍스트 확인

1. `gh issue list`로 열린 이슈 현황을 확인한다
2. 사용자가 특정 작업을 지목하면 그 작업. "다음 작업"이면 정해진 순서 문서가 없으므로 열린 이슈 중 우선순위가 명확한 것을 고르거나, 애매하면 사용자에게 확인한다
3. 부분 재실행(같은 작업 보완 요청)이면 이전 결과·QA 리포트(`_workspace/`)를 에이전트 프롬프트에 포함

### Phase 1: Pre-게이트 (착수 전 — 하나라도 실패하면 착수 금지)

1. **레인·경계 배정**: 작업이 단일 레인에 속하는가? 두 레인을 걸치면 레인별로 분할해 순차 위임
2. **계약 영향 체크 (최중요 게이트)**: 이 작업이 CLAUDE.md의 경계 원칙(API 경로·요청/응답 스키마·`policy_announcement` DDL)을 변경하는가?
   → 변경한다면 **영향받는 양쪽 코드(예: ai-engine 라우터 + `AiEngineClient.java`)를 같은 사이클에 위임**한다. 계약 문서가 없으므로 코드 자체를 단일 소스로 삼아 드리프트를 막는다
3. **DoD 확정**: §3 행의 "남은 작업"을 검증 가능한 완료 조건으로 구체화. DoD 없이 착수 금지

### Phase 2: 위임

컨텍스트 패킷을 조립해 해당 에이전트를 호출한다:

```
패킷 = 작업 ID·DoD + §1 경계·금지 + §2 관련 계약 슬라이스(전체 아님)
     + 대상 파일 경로 + (프롬프트·RAG 작업이면) 골든 I/O 예시 1~2쌍
     + (재실행이면) 이전 QA 리포트 경로
```

저장소 전체를 주지 않는다. 상대 레인은 인터페이스 파일만 (예: ai-dev에게 `AiEngineClient.java`).

### Phase 3: Post-게이트 (완료 선언 전)

1. **경계위반 스캔** (오케스트레이터가 직접, grep 수준): ai-engine의 비즈니스 테이블 접근 / Spring의 anthropic 의존성 / web의 `:8000` 호출 / 업종·지역 하드코딩 / 신청서 자동 제출 / `ddl-auto` 드리프트
2. **qa-verifier 위임** (저작·검증 분리 — 저작 에이전트의 자기승인 금지): DoD 충족 + 경계면 교차 검증. 리포트는 `_workspace/qa_{작업ID}.md`
3. **마일스톤 E2E**: S4·S5·S6 등 크리티컬 패스 조각 착지 직후 qa-verifier에게 e2e-verify 스킬 기반 회귀를 요청 (`POST /api/agent/check/1`)

### Phase 4: 상태 갱신·보고

1. QA 통과 시 관련 GitHub 이슈를 닫거나 진행 상태를 갱신한다
2. QA 실패 시 실패 항목(파일:라인)을 담당 에이전트에 재위임 (최대 2회, 이후 사용자 보고)
3. 사용자에게 요약: 완료 작업 / QA 결과 / 다음 순서 작업

## 에러 핸들링

| 상황 | 전략 |
| --- | --- |
| 에이전트가 계약 변경 필요 보고 | 구현 중단 → Phase 1-2 계약 절차로 회귀 (문서 먼저) |
| 에이전트 실패/막힘 | 1회 재시도(실패 로그 첨부), 재실패 시 사용자에게 원인·선택지 보고 |
| QA-수정 2회 반복 후에도 실패 | 루프 중단, 사용자에게 상충 지점 보고 (무한 루프 금지) |
| 환경 이슈 (키·도커 미기동) | 해당 검증을 "미검증"으로 명시하고 진행 — 조용히 통과 처리 금지 |

## 테스트 시나리오

**정상**: "다음 작업 진행해" → Phase 0에서 §7 첫 미완(①·S4) 선택 → Pre-게이트 통과(계약 무영향, DoD=latestMetric 매핑+시드로 트리거 발동) → backend-dev 위임 → qa-verifier가 SQL↔DDL 교차 검증 + E2E 게이트 1~2 → §3 상태 갱신 → 보고.

**에러**: ai-dev가 "/matching 응답에 evidence 필드 추가 필요" 보고 → Phase 2 중단 → §2-2 문서 수정(evidence 추가) → ai-dev(응답 생성)와 backend-dev(`AiEngineClient`·`funding_match` 저장) 같은 사이클 위임 → qa-verifier가 양쪽 동시 읽기로 계약 일치 확인.
